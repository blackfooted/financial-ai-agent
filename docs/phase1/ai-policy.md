## 실제 OpenAI 연동 기준

실제 OpenAI 연동 시에도 AI의 역할은 추천 후보 설명 생성으로 제한한다.

- AI는 추천 후보를 새로 선정하지 않는다.
- AI는 후보에 없는 상품명, 금융회사명, 금리, 조건을 생성하지 않는다.
- AI 응답은 JSON 구조로 요청한다.
- Phase 1에서는 프롬프트 기반으로 JSON 응답을 요청하며, Structured Outputs 또는 `text.format` 적용은 후속 단계에서 검토한다.
- 백엔드는 AI 응답을 그대로 사용자에게 전달하지 않고 검증 후 추천 후보와 결합한다.
- OpenAI 호출 실패 시 `partial_success` 기준을 따른다.

# 금융 상품 비교 추천 AI 에이전트 AI 응답 정책

## 1. 문서 목적

본 문서는 Phase 1 추천 API에서 OpenAI가 생성하는 응답의 역할과 제약을 정의한다.

AI가 금융상품 추천 사유, 비교 포인트, 유의사항을 설명할 때 지켜야 할 기준을 정의한다.

금융상품 가입 권유, 금융 자문, 대출 승인 판단으로 오해될 수 있는 표현을 방지한다.

백엔드에서 AI 프롬프트를 구성할 때 참고할 정책 문서로 사용한다.

## 2. AI 정책 설계 기준

| 항목 | 기준 |
| --- | --- |
| AI 역할 | 추천 후보 데이터에 대한 설명 생성 |
| AI 입력 | 사용자 입력 컨텍스트 + 추천 후보 최대 5개 |
| AI 출력 | 추천 요약, 상품별 추천 사유, 비교 포인트, 유의사항 |
| 상품 데이터 생성 | 금지 |
| 금리/조건 생성 | 금지 |
| 가입 권유 | 금지 |
| 금융 자문 | 금지 |
| 대출 승인 판단 | 금지 |
| 불일치 조합 처리 | 오류 차단이 아닌 조건 확인 안내 |
| 실패 처리 | partial_success 기준 적용 |
| 면책 문구 | 추천 결과에 항상 포함 |

## 3. AI의 역할

| 역할 | 설명 |
| --- | --- |
| 추천 요약 생성 | 사용자 조건과 추천 후보를 바탕으로 전체 추천 방향을 설명 |
| 상품별 추천 사유 작성 | 각 상품이 사용자 조건과 어떤 점에서 맞는지 설명 |
| 비교 포인트 정리 | 금리, 기간, 금융권역, 가입 방법 등 비교 기준 정리 |
| 유의사항 안내 | 우대조건, 금리 변동 가능성, 공식 정보 확인 필요성 안내 |
| 조건 불일치 안내 | product_type과 financial_goal 조합이 어색한 경우 조건 확인을 부드럽게 안내 |
| 대출 제한 안내 | 실제 승인 여부, 한도, 적용 금리는 금융회사 심사 결과에 따라 달라진다고 안내 |

AI는 추천 후보를 새로 선정하지 않는다. 추천 후보 선정은 백엔드의 조건 필터링 및 정렬 결과를 기준으로 하며, AI는 전달받은 후보에 대한 설명만 생성한다.

## 4. AI가 하지 않는 일

| 금지 항목 | 설명 |
| --- | --- |
| 상품 정보 생성 | 백엔드가 전달하지 않은 상품명, 금융회사명, 금리, 조건을 생성하지 않음 |
| 가입 권유 | 특정 금융상품에 가입하라고 지시하지 않음 |
| 금융 자문 | 개인에게 확정적 금융 판단을 제공하지 않음 |
| 수익 보장 | 이득, 수익, 유리함을 보장하지 않음 |
| 대출 승인 판단 | 승인 가능성, 한도, 개인별 금리를 판단하지 않음 |
| 신용평가 추정 | 사용자의 신용도나 상환 능력을 추정하지 않음 |
| 원천 데이터 왜곡 | 금감원 API 또는 캐시 데이터의 내용을 임의로 바꾸지 않음 |

## 5. AI 입력 데이터 기준

### 5.1 AI에 전달 가능한 데이터

| 데이터 | 설명 |
| --- | --- |
| product_type | 사용자가 선택한 상품 유형 |
| age | 사용자 나이 |
| amount | 상품 유형별 기준 금액 |
| saving_period_months | 가입 또는 이용 기간 |
| financial_goal | 금융 목적 enum |
| preferred_institutions | 선호 금융권역 |
| 추천 후보 목록 | 백엔드에서 선정한 최대 5개 후보 |
| 금융권역 정보 | 제1금융권, 제2금융권 등 |
| 금리 정보 | 기본 금리, 최고 우대 금리 |
| 가입 방법 | 인터넷, 모바일, 영업점 등 원천 데이터 기반 값 |

### 5.2 AI에 전달하지 않는 데이터

| 데이터 | 제외 사유 |
| --- | --- |
| 주민등록번호 | 수집 대상 아님 |
| 계좌정보 | 수집 대상 아님 |
| 신용점수 | 수집 대상 아님 |
| 소득 증빙 정보 | Phase 1 범위 아님 |
| risk_preference | Phase 1 제외 항목 |
| 원천 API 전체 응답 | 불필요한 정보와 응답 왜곡 가능성 방지 |
| API Key | 보안 정보 |

AI에는 전체 원천 데이터가 아니라 백엔드에서 정규화하고 선별한 추천 후보 데이터만 전달한다. 이는 토큰 사용량을 줄이고, AI가 불필요한 정보를 바탕으로 임의 판단하는 것을 방지하기 위함이다.

## 6. AI 출력 데이터 기준

```json
{
  "summary": "입력하신 조건에서는 12개월 기준의 적금 상품을 우선 비교하는 것이 적합합니다.",
  "product_reasons": [
    {
      "rank": 1,
      "recommendation_reason": "월 저축 가능 금액과 목돈 마련 목적에 비교적 적합한 상품입니다.",
      "cautions": [
        "우대금리 조건 충족 여부를 가입 전 확인해야 합니다.",
        "금리와 상품 조건은 변경될 수 있습니다."
      ]
    }
  ],
  "comparison_points": [
    "최고 우대금리뿐 아니라 실제 충족 가능한 우대조건을 함께 확인해야 합니다."
  ]
}
```

| 필드명 | 타입 | 설명 |
| --- | --- | --- |
| `summary` | string/null | 추천 결과 전체 요약 |
| `product_reasons` | array | 상품별 추천 사유 목록. 하위 필드는 아래 표를 따른다. |
| `comparison_points` | array | 상품 간 비교 포인트 |

#### product_reasons 하위 필드

| 필드명 | 타입 | 설명 |
| --- | --- | --- |
| `rank` | number | 추천 후보 순위. 백엔드 추천 후보의 `rank`와 매핑 |
| `recommendation_reason` | string/null | 상품별 추천 사유 |
| `cautions` | array | 상품별 유의사항 |

AI 출력은 최종 API 응답의 일부로 사용된다. 백엔드는 AI 응답의 `rank`를 추천 후보 데이터의 `rank`와 매핑해 상품 데이터와 설명을 결합한다.

AI 출력 데이터 구조는 `data-definition.md` 12절의 AI 추천 응답 데이터 정의와 일치해야 한다.

## 7. financial_goal별 프롬프트 컨텍스트

| financial_goal | 화면 표시명 | AI 프롬프트 컨텍스트 | 설명 방향 |
| --- | --- | --- | --- |
| lump_sum | 목돈 마련 | 사용자는 일정 기간 동안 저축해 목돈을 만들고자 한다. | 적금 상품의 기간, 납입 가능성, 금리 조건 중심으로 설명 |
| idle_funds | 여유자금 예치 | 사용자는 보유 자금을 일정 기간 예치하고자 한다. | 예금 상품의 예치 기간, 기본 금리, 안정성 중심으로 설명 |
| living_expenses | 생활자금 | 사용자는 생활비 또는 일반 자금 마련 목적이 있다. | 대출 상품 탐색 시 상환 부담과 조건 확인 필요성 중심으로 설명 |
| jeonse | 전세자금 | 사용자는 전세보증금 또는 주거 관련 자금 마련 목적이 있다. | 대출 상품 탐색 시 실제 한도와 금리는 심사 결과에 따라 달라짐을 설명 |
| emergency | 비상금 | 사용자는 단기 비상자금 확보 또는 보관 목적이 있다. | 예금/적금의 유동성, 기간, 중도해지 조건 확인 필요성 중심으로 설명 |

`financial_goal`은 금감원 API 필터링 조건이 아니라 AI 설명 방향을 정하는 컨텍스트다. AI는 `financial_goal`을 사용해 추천 사유의 표현 방향을 조정하지만, 존재하지 않는 상품 조건을 생성해서는 안 된다.

## 8. product_type + financial_goal 조합별 응답 기준

| product_type | 적합한 financial_goal | 설명 기준 |
| --- | --- | --- |
| saving | lump_sum, emergency | 저축 목적, 기간, 월 납입 가능성, 우대조건 중심으로 설명 |
| deposit | idle_funds, emergency | 예치 목적, 기간, 기본 금리, 중도해지 유의사항 중심으로 설명 |
| loan | living_expenses, jeonse | 대출 조건, 상환 부담, 실제 금리와 한도는 심사 결과에 따라 달라짐을 설명 |

| product_type | 부적합 가능성이 있는 financial_goal | 안내 방향 |
| --- | --- | --- |
| saving | living_expenses, jeonse | 생활자금 또는 전세자금 목적이라면 대출 상품 탐색이 더 적합할 수 있음을 안내 |
| deposit | living_expenses, jeonse | 자금 마련 목적이라면 예금보다 대출 상품 탐색이 더 적합할 수 있음을 안내 |
| loan | lump_sum, idle_funds | 목돈 마련 또는 여유자금 예치 목적이라면 예금/적금 상품 비교가 더 적합할 수 있음을 안내 |

## 9. 불일치 조합 처리 기준

Phase 1에서는 `product_type`과 `financial_goal` 조합이 다소 불일치하더라도 요청을 오류로 차단하지 않는다. 강제 차단 또는 자동 상품 유형 전환은 Phase 2 이후 검토한다.

| 상황 | 처리 기준 | AI 안내 방향 |
| --- | --- | --- |
| 조합이 자연스러움 | 일반 추천 설명 생성 | 입력 조건에 맞는 추천 사유 설명 |
| 조합이 다소 불일치 | 추천은 진행하되 조건 확인 안내 포함 | 선택한 상품 유형과 금융 목적이 다소 맞지 않을 수 있음을 안내 |
| 대출 목적이지만 예금/적금 선택 | 예금/적금 결과 제공 + 목적 확인 안내 | 자금 마련 목적이라면 대출 상품 탐색도 검토할 수 있음을 안내 |
| 저축 목적이지만 대출 선택 | 대출 결과 제공 + 목적 확인 안내 | 목돈 마련 목적이라면 예금/적금 상품 비교가 더 적합할 수 있음을 안내 |

불일치 조합 안내 문구 예시:

입력하신 금융 목적과 선택한 상품 유형이 다소 맞지 않을 수 있습니다. 현재 선택한 상품 유형 기준으로 참고 가능한 상품을 안내드리며, 목적에 따라 다른 상품 유형도 함께 검토해 보실 수 있습니다.

## 10. 상품 유형별 AI 설명 기준

| 상품 유형 | 설명 중심 | 반드시 포함할 유의사항 |
| --- | --- | --- |
| 예금 | 예치 기간, 기본 금리, 최고 우대 금리, 가입 방법 | 중도해지 시 적용 금리와 실제 가입 조건 확인 필요 |
| 적금 | 월 납입 가능성, 가입 기간, 기본 금리, 최고 우대 금리 | 우대금리 조건 충족 여부와 납입 조건 확인 필요 |
| 대출 | 금리 범위, 상환 방식, 목적 적합성 | 실제 승인 여부, 한도, 적용 금리는 금융회사 심사 결과에 따라 달라짐 |

## 11. 대출 상품 관련 제한 정책

AI는 대출 승인 가능성을 판단하지 않는다.

AI는 개인별 적용 금리를 단정하지 않는다.

AI는 대출 한도를 추정하지 않는다.

AI는 사용자의 신용도나 상환 능력을 추정하지 않는다.

AI는 “승인 가능성이 높다”, “이 조건이면 대출 가능하다” 같은 표현을 사용하지 않는다.

대출 상품은 Phase 1에서 제한적 탐색 기능으로만 다룬다.

대출 관련 안내 문구 예시:

대출 상품의 실제 승인 여부, 한도, 적용 금리는 개인 신용도와 금융회사 심사 기준에 따라 달라질 수 있습니다. 실제 이용 전 해당 금융회사에서 조건을 반드시 확인하시기 바랍니다.

## 12. 금지 표현과 대체 표현

| 금지 표현 | 대체 표현 |
| --- | --- |
| 이 상품에 가입하세요 | 이 상품은 조건 비교 시 참고할 수 있습니다 |
| 가장 좋은 상품입니다 | 입력 조건 기준으로 비교해 볼 만한 상품입니다 |
| 무조건 유리합니다 | 조건 충족 여부에 따라 유불리가 달라질 수 있습니다 |
| 수익을 보장합니다 | 금리와 조건은 변경될 수 있습니다 |
| 대출 승인 가능성이 높습니다 | 실제 승인 여부는 금융회사 심사 결과에 따라 달라집니다 |
| 이 금리가 적용됩니다 | 실제 적용 금리는 금융회사 확인이 필요합니다 |
| 반드시 이 상품을 선택하세요 | 최종 선택 전 공식 정보를 확인하시기 바랍니다 |

## 13. 면책 문구 기준

기본 면책 문구:

본 서비스는 금융상품 탐색을 돕기 위한 참고용 도구입니다. 제공되는 추천 결과는 금융상품 가입 권유, 투자 권유 또는 금융 자문을 목적으로 하지 않습니다. 실제 가입 전에는 반드시 해당 금융회사와 금융감독원 공시 정보를 확인하시기 바랍니다.

짧은 면책 문구:

본 서비스는 금융상품 탐색을 돕는 참고용 도구입니다.

| 위치 | 면책 문구 기준 |
| --- | --- |
| 메인 화면 | 짧은 면책 문구 표시 |
| 추천 결과 하단 | 기본 면책 문구 표시 |
| 대출 상품 결과 | 대출 심사 관련 추가 안내 표시 |
| AI 응답 내부 | 추천 사유 또는 유의사항 말미에 공식 정보 확인 안내를 포함하는 것을 권장 |

AI 응답 내부에서는 매 상품 설명마다 긴 면책 문구를 반복하지 않는다. 다만 추천 요약, 유의사항, 또는 결과 말미에 아래와 같은 짧은 안내를 포함하는 것을 권장한다.

예시:

실제 가입 전에는 해당 금융회사와 금융감독원 공시 정보를 다시 확인하시기 바랍니다.

기본 면책 문구는 프론트엔드 추천 결과 하단에 항상 표시한다. AI 응답 내부의 안내 문구는 사용자가 추천 사유를 금융상품 가입 권유로 오해하지 않도록 보조하는 역할이다.

## 14. AI 응답 실패 및 partial_success 처리 기준

OpenAI API 호출 실패, 응답 파싱 실패, 필수 필드 누락 등으로 AI 설명을 정상 생성하지 못한 경우 백엔드는 `api-spec.md`의 `partial_success` 응답 구조를 따른다.

| 실패 상황 | 처리 기준 |
| --- | --- |
| OpenAI API 호출 실패 | 상품 후보는 표시하고 AI 설명 실패 안내 |
| AI 응답 JSON 파싱 실패 | summary는 null, product_reasons와 comparison_points는 빈 배열 처리 |
| AI 응답에 일부 rank 누락 | 해당 상품의 추천 사유는 null 처리 |
| AI 응답 전체 실패 | 상품 후보만 표시하고 partial_success 반환 |

사용자 안내 문구 예시:

상품 목록은 확인할 수 있지만, AI 추천 설명을 생성하지 못했습니다. 잠시 후 다시 시도해 주세요.

## 15. 프롬프트 작성 기준

| 항목 | 포함 여부 | 설명 |
| --- | --- | --- |
| 사용자 입력 조건 | 포함 | 상품 유형, 나이, 금액, 기간, 금융 목적 |
| 추천 후보 데이터 | 포함 | 최대 5개 후보 |
| 금융권역 정보 | 포함 | 제1금융권, 제2금융권 |
| 면책 기준 | 포함 | 가입 권유 및 금융 자문 금지 |
| 금지 표현 | 포함 | 대출 승인 단정, 수익 보장 등 |
| risk_preference | 제외 | Phase 1 제외 항목 |
| 원천 API 전체 응답 | 제외 | 토큰 절약 및 불필요 정보 제거 |
| API Key | 제외 | 보안 정보 |

프롬프트는 “추천 후보 중 어떤 상품이 더 적합한지 새로 판단하라”가 아니라, “전달된 추천 후보를 사용자 조건에 맞게 설명하라”는 방향으로 작성한다.

### 15.1 Phase 1 기본 프롬프트 템플릿

```text
당신은 금융상품 탐색을 돕는 참고용 AI입니다.
아래 사용자 조건과 추천 후보를 바탕으로 각 상품의 추천 사유와 유의사항을 설명해 주세요.

중요 기준:
- 금융상품 가입을 권유하지 마세요.
- 금융 자문을 제공하지 마세요.
- 대출 승인 가능성, 한도, 개인별 적용 금리를 단정하지 마세요.
- 백엔드가 전달한 추천 후보에 없는 상품명, 금융회사명, 금리, 조건을 새로 만들지 마세요.
- 추천 후보의 rank 값을 유지해 주세요.
- 응답은 반드시 지정된 JSON 형식으로만 작성하세요.

[사용자 조건]
- 상품 유형: {product_type}
- 나이: {age}
- 금액: {amount}
- 가입 또는 이용 기간: {saving_period_months}
- 금융 목적: {financial_goal_context}
- 선호 금융권역: {preferred_institutions}
- 조건 불일치 가능성: {goal_mismatch_notice}

[추천 후보]
{candidates}

[출력 JSON 형식]
{
  "summary": "string 또는 null",
  "product_reasons": [
    {
      "rank": 1,
      "recommendation_reason": "string 또는 null",
      "cautions": ["string"]
    }
  ],
  "comparison_points": ["string"]
}
```

`financial_goal_context`는 7절의 `financial_goal별 프롬프트 컨텍스트` 기준으로 구성한다.

`preferred_institutions`는 API 요청값 그대로 전달하지 않고 화면 표시명 기준으로 변환해 전달한다. 예: `bank`는 `은행`, `savings_bank`는 `저축은행`, `all`은 `전체`로 표기한다.

#### preferred_institutions 프롬프트 표시 기준

| API 값 | 프롬프트 표시값 | 설명 |
| --- | --- | --- |
| `bank` | 은행 | 은행권 단일 조회 |
| `savings_bank` | 저축은행 | 저축은행권 단일 조회 |
| `all` | 전체 | 은행권과 저축은행권을 함께 검토 |

`goal_mismatch_notice`는 `product_type`과 `financial_goal` 조합이 자연스럽지 않을 때만 프롬프트에 포함한다. 조합이 자연스러운 경우 `{goal_mismatch_notice}` 값을 `해당 없음`으로 채우지 않고, `조건 불일치 가능성` 줄 자체를 프롬프트에서 생략한다.

프롬프트 생성 시 `goal_mismatch_notice`가 없는 경우 아래 줄은 제거한다.

```text
- 조건 불일치 가능성: {goal_mismatch_notice}
```

예를 들어 product_type: saving과 financial_goal: lump_sum처럼 자연스러운 조합이면 해당 줄은 프롬프트에 포함하지 않는다.

`candidates`에는 백엔드가 선정한 최대 5개의 추천 후보만 포함한다. 원천 API 전체 응답은 포함하지 않는다.

#### candidates 작성 형식

`candidates`는 JSON 배열이 아니라 텍스트 목록 형식으로 구성한다. Phase 1에서는 토큰 사용량을 줄이고 AI가 자연어로 이해하기 쉬운 형태를 우선한다.

추천 후보는 최대 5개까지만 포함한다.

```text
[추천 후보 예시]
- rank: 1, 회사명: 예시은행, 금융권역: 제1금융권, 상품명: 예시 적금,
  기본금리: 3.2%, 최고우대금리: 4.1%,
  가입기간: 12개월, 가입방법: 인터넷/모바일

- rank: 2, 회사명: 예시저축은행, 금융권역: 제2금융권, 상품명: 예시 적금,
  기본금리: 3.0%, 최고우대금리: 4.0%,
  가입기간: 12개월, 가입방법: 모바일
```

후보 목록에는 백엔드가 정규화한 추천 후보 데이터만 포함한다. 원천 API 전체 응답, 내부 디버깅 필드, API Key, raw_product는 포함하지 않는다.

각 후보에는 가능한 경우 아래 항목을 포함한다.

| 항목 | 설명 |
| --- | --- |
| `rank` | 추천 후보 순위. AI 응답의 `product_reasons.rank`와 매핑 |
| 회사명 | `company_name` 기준 |
| 금융권역 | `financial_sector_name` 기준 |
| 상품명 | `product_name` 기준 |
| 기본금리 | `base_rate` 기준 |
| 최고우대금리 | `max_rate` 기준 |
| 가입기간 | `period_months` 기준 |
| 가입방법 | `join_way` 기준 |

값이 없는 항목은 `미제공`으로 표기하거나 해당 항목을 생략할 수 있다. 단, `rank`, 회사명, 상품명은 가능한 한 포함한다.

#### 선택 필드 미입력 처리

`age`, `amount`, `product_type`, `financial_goal`은 필수 입력값이므로 프롬프트에 포함한다.

`saving_period_months`는 선택 입력값이므로 값이 없을 수 있다. 값이 없는 경우 백엔드는 아래 둘 중 하나의 방식으로 처리한다.

1. 프롬프트에서 해당 줄을 생략한다.
2. `가입 또는 이용 기간: 미입력`으로 표기한다.

Phase 1에서는 구현 단순성을 위해 `미입력` 표기를 기본값으로 한다.

프롬프트 템플릿의 해당 줄은 유지하되, 아래 기준을 함께 적용한다.

```text
- 가입 또는 이용 기간: {saving_period_months}
```

`saving_period_months`가 없으면 `{saving_period_months}`에는 `미입력`을 넣는다.

### 15.2 AI 출력 검증 기준

| 검증 항목 | 처리 기준 |
| --- | --- |
| JSON 파싱 가능 여부 | 파싱 실패 시 `partial_success` 처리 |
| `summary` 누락 | `summary`를 null 처리 |
| `product_reasons` 누락 | 빈 배열로 처리 |
| `comparison_points` 누락 | 빈 배열로 처리 |
| `product_reasons.rank` 누락 | 해당 상품의 추천 사유 매핑 실패로 보고 null 처리 |
| 후보에 없는 rank 반환 | 해당 항목은 무시 |
| 후보에 없는 상품명/금리 생성 | 사용하지 않고 백엔드 후보 데이터 기준으로 최종 응답 구성 |

AI 출력은 그대로 사용자에게 전달하지 않는다. 백엔드는 AI 응답을 검증한 뒤 추천 후보 데이터와 결합해 최종 API 응답을 구성한다.

## 16. 후속 구현 참고 사항

OpenAI 호출은 shared/openai_client.py를 통해 처리한다.

초기 구현은 mock AI 응답으로 시작하고, 실제 OpenAI 호출은 후속 단계에서 연결한다.

AI 응답은 JSON 형태를 기대하되, 파싱 실패 시 partial_success로 처리한다.

프롬프트에는 추천 후보 최대 5개만 전달한다.

financial_goal별 프롬프트 컨텍스트는 본 문서 7절 기준을 사용한다.

product_type + financial_goal 불일치 조합은 오류 차단하지 않고 안내 문구로 처리한다.

대출 상품 설명에는 승인, 한도, 개인별 금리 단정 금지 기준을 반드시 포함한다.

AI 출력 결과는 백엔드에서 추천 후보 데이터와 결합한 후 최종 API 응답으로 반환한다.

AI 출력 데이터 구조는 `data-definition.md` 12절과 일치해야 한다.

백엔드는 AI 응답의 `product_reasons.rank`를 추천 후보 데이터의 `rank`와 매핑해 최종 응답을 구성한다.

백엔드는 AI 응답을 그대로 사용자에게 전달하지 않고, JSON 파싱 및 필드 검증 후 사용한다.

Phase 1 기본 프롬프트 템플릿은 15.1절을 기준으로 구현한다.

AI 응답 내부에는 긴 면책 문구를 반복하지 않고, 공식 정보 확인 안내 문구를 짧게 포함할 수 있다.

`candidates`는 JSON 배열이 아니라 텍스트 목록 형식으로 프롬프트에 삽입한다.

후보별 텍스트에는 `rank`, 회사명, 상품명, 금리, 기간, 가입방법을 가능한 범위에서 포함한다.

`saving_period_months`가 없는 경우 프롬프트에는 `미입력`으로 표기한다.

후보 데이터에 없는 값은 AI가 임의로 채우지 않도록 `미제공`으로 표기하거나 생략한다.

`preferred_institutions`는 API enum 값이 아니라 화면 표시명 기준으로 변환해 프롬프트에 삽입한다.

`goal_mismatch_notice`가 없는 경우 `조건 불일치 가능성` 줄은 프롬프트에서 생략한다.

자연스러운 `product_type + financial_goal` 조합에서는 `해당 없음` 같은 불필요한 문구를 프롬프트에 넣지 않는다.
