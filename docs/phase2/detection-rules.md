# Phase 2 탐지 룰 정의서

## 문서 목적

본 문서는 Phase 2 룰 기반 의심거래 탐지 기준을 정의한다. Step 3 백엔드 mock API 구현 시 룰 코드, 상세 조건, `risk_score`, 위험도 매핑을 임의로 해석하지 않도록 기준을 명확히 한다.

## 탐지 룰 설계 원칙

- 룰 기반 탐지는 의심 후보를 선별하는 기준일 뿐 최종 판단이 아니다.
- AI는 룰 기반 탐지 결과를 설명하는 리포트 초안을 생성한다.
- 최종 상태 변경은 담당자가 수행한다.
- 샘플 데이터에서는 이해하기 쉬운 기준값을 사용하고, real 데이터 전환 시 운영 정책에 맞게 조정한다.
- 룰 코드는 API 응답과 리포트 생성 입력에서 동일하게 사용한다.

## 탐지 룰 코드 정의

| 룰 코드 | 설명 | 위험도 기여 |
|---|---|---|
| HIGH_AMOUNT | 거래 금액이 기준값을 초과한 경우 | high |
| NEW_DEVICE | 기존 거래 이력에 없는 기기에서 거래가 발생한 경우 | high |
| REPEAT_TX | 단시간 내 동일 고객 또는 계좌에서 반복 거래가 발생한 경우 | medium |
| ODD_HOUR | 평소 이용 시간대와 다른 심야 시간대에 거래가 발생한 경우 | low |
| LOCATION_MISMATCH | 짧은 시간 내 물리적으로 이동하기 어려운 위치에서 거래가 발생한 경우 | medium |

## 룰별 상세 조건

| 룰 코드 | 상세 조건 예시 | 담당자 확인 포인트 |
|---|---|---|
| HIGH_AMOUNT | 거래 금액이 3,000,000원 이상인 경우 | 최근 평균 거래 금액, 수취 계좌, 거래 목적을 확인한다. |
| NEW_DEVICE | 동일 고객의 기존 `device_id` 목록에 없는 기기에서 거래한 경우 | 신규 기기 등록 이력과 본인 사용 여부를 확인한다. |
| REPEAT_TX | 동일 고객이 10분 이내 3회 이상 거래한 경우 | 분할 이체, 자동화 시도, 동일 수취인 여부를 확인한다. |
| ODD_HOUR | 00:00~05:00 사이 거래한 경우 | 고객 평소 이용 시간대와 거래 목적을 확인한다. |
| LOCATION_MISMATCH | 1시간 이내 서로 다른 국가 또는 이동 불가능한 지역에서 거래한 경우 | 직전 거래 위치, 접속 IP, 기기 정보를 함께 확인한다. |

## risk_score 산정 기준

기본 점수:
- 탐지 룰 1개 적용: 30점
- 탐지 룰 2개 이상 적용: 첫 번째 룰 30점 + 추가 룰당 20점
- HIGH_AMOUNT 포함 시: +10점
- NEW_DEVICE 포함 시: +10점
- 최대 점수는 100점으로 제한한다.

## 위험도 매핑 기준

| risk_score | 위험도 코드값 | 화면 표시명 |
|---|---|---|
| 0~49점 | low | 하 |
| 50~79점 | medium | 중 |
| 80~100점 | high | 상 |

## 룰 조합 예시

| 조합 | 산정 방식 | risk_score | 위험도 |
|---|---|---|---|
| 탐지 룰 없음 | 0점 | 0 | low(하) |
| HIGH_AMOUNT | 30점 + HIGH_AMOUNT 10점 | 40 | low(하) |
| ODD_HOUR | 30점 | 30 | low(하) |
| HIGH_AMOUNT, NEW_DEVICE | 30점 + 추가 룰 20점 + HIGH_AMOUNT 10점 + NEW_DEVICE 10점 | 70 | medium(중) |
| REPEAT_TX, ODD_HOUR | 30점 + 추가 룰 20점 | 50 | medium(중) |
| HIGH_AMOUNT, NEW_DEVICE, LOCATION_MISMATCH | 30점 + 추가 룰 40점 + HIGH_AMOUNT 10점 + NEW_DEVICE 10점 | 90 | high(상) |

## 향후 확장 고려사항

- 기준 금액, 시간 범위, 반복 횟수는 운영 정책에 따라 설정값으로 분리할 수 있다.
- 고객별 평균 거래 금액, 선호 채널, 거주 지역 등 프로파일 기반 룰을 추가할 수 있다.
- 룰 변경 이력과 적용 버전을 저장하여 탐지 결과의 재현 가능성을 보장해야 한다.
- 실제 운영계에서는 담당자 권한, 감사 로그, 상태 역전이 제한을 함께 검토해야 한다.
