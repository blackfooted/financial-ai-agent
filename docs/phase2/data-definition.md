# Phase 2 데이터 정의서

## 문서 목적

본 문서는 Phase 2 "FDS 의심거래 탐지 및 담당자 검토 리포트 자동화 에이전트"에서 사용할 샘플 거래 데이터, 탐지 결과, 탐지 룰, 위험도, 검토 상태의 정의를 정리한다. 개발 전 데이터 구조와 용어를 통일하여 API 설계와 화면 설계의 기준으로 활용한다.

## 샘플 거래 데이터 설계 원칙

- 포트폴리오 목적에 맞게 실제 FDS 업무 흐름을 이해할 수 있는 수준의 거래 맥락을 포함한다.
- 실제 고객 개인정보, 계좌번호, IP, 기기정보는 사용하지 않고 가상의 값만 사용한다.
- 탐지 룰 검증이 가능하도록 정상 거래와 의심 거래 시나리오를 함께 포함한다.
- 담당자가 거래 금액, 채널, 위치, 시간, 기기 정보를 보고 검토할 수 있도록 필드를 구성한다.
- 향후 real 데이터 확장을 고려해 식별자, 상태값, 시간 필드는 일관된 이름을 사용한다.

## 거래 데이터 필드 정의

| 필드명 | 타입 | 설명 | 예시 |
|---|---|---|---|
| id | string | 내부 데이터 식별자 | tx-row-001 |
| transaction_id | string | 거래 고유 식별자 | TX202605180001 |
| customer_id | string | 고객 식별자. 샘플에서는 익명화된 가상 ID를 사용한다. | CUST-1001 |
| account_id | string | 계좌 식별자. 샘플에서는 가상 ID를 사용한다. | ACC-2001 |
| transaction_type | string | 거래 유형 | transfer |
| amount | number | 거래 금액 | 3500000 |
| currency | string | 통화 코드 | KRW |
| channel | string | 거래 발생 채널 | mobile |
| device_id | string | 거래에 사용된 기기 식별자 | DEV-SEOUL-01 |
| ip_address | string | 거래 요청 IP. 샘플에서는 문서용 가상 IP를 사용한다. | 192.0.2.10 |
| location | string | 거래 발생 위치 또는 추정 지역 | Seoul |
| transaction_time | string | 거래 발생 시각 | 2026-05-18T09:30:00+09:00 |
| created_at | string | 샘플 데이터 생성 시각 | 2026-05-18T09:31:00+09:00 |

## 탐지 결과 필드 정의

| 필드명 | 타입 | 설명 | 예시 |
|---|---|---|---|
| id | string | 탐지 결과 식별자 | det-001 |
| transaction_id | string | 탐지 대상 거래 식별자 | TX202605180001 |
| detected_rules | string[] | 적용된 의심거래 탐지 룰 목록 | ["HIGH_AMOUNT", "NEW_DEVICE"] |
| risk_level | string | 위험도 코드값. `low`, `medium`, `high` 중 하나를 사용한다. | medium |
| risk_score | number | 위험도 점수. 0에서 100 사이의 값을 사용한다. | 70 |
| reason_summary | string | 담당자가 이해할 수 있는 탐지 사유 요약 | 고액 이체와 신규 기기 사용이 동시에 확인됨 |
| review_status | string | 담당자 검토 상태 | 미확인 |

| reviewed_at | string | 검토 완료 또는 상태 변경 시각 | 2026-05-18T10:10:00+09:00 |
| created_at | string | 탐지 결과 생성 시각 | 2026-05-18T09:32:00+09:00 |
| updated_at | string | 탐지 결과 최종 수정 시각 | 2026-05-18T10:10:00+09:00 |

## 탐지 룰 코드 정의

| 룰 코드 | 설명 | 위험도 기여 |
|---|---|---|
| HIGH_AMOUNT | 거래 금액이 기준값을 초과한 경우 | high |
| NEW_DEVICE | 기존 거래 이력에 없는 기기에서 거래가 발생한 경우 | high |
| REPEAT_TX | 단시간 내 동일 고객 또는 계좌에서 반복 거래가 발생한 경우 | medium |
| ODD_HOUR | 평소 이용 시간대와 다른 심야 시간대에 거래가 발생한 경우 | low |
| LOCATION_MISMATCH | 짧은 시간 내 물리적으로 이동하기 어려운 위치에서 거래가 발생한 경우 | medium |

## risk_score 산정 기준

기본 점수:
- 탐지 룰 1개 적용: 30점
- 탐지 룰 2개 이상 적용: 첫 번째 룰 30점 + 추가 룰당 20점
- HIGH_AMOUNT 포함 시: +10점
- NEW_DEVICE 포함 시: +10점
- 최대 점수는 100점으로 제한한다.

위험도 매핑:
- 0~49점: `low(하)`
- 50~79점: `medium(중)`
- 80~100점: `high(상)`

## 위험도 기준

| 위험도 코드값 | 화면 표시명 | 기준 | 담당자 검토 관점 |
|---|---|---|---|
| low | 하 | 미탐지 또는 단일 경미 룰이 탐지되었고 risk_score가 0~49점인 경우 | 일반 모니터링 대상으로 확인한다. |
| medium | 중 | 복수의 탐지 룰이 적용되었거나 risk_score가 50~79점인 경우 | 담당자가 거래 상세와 고객 맥락을 확인한다. |
| high | 상 | 주요 위험 신호가 복합적으로 확인되었거나 risk_score가 80~100점인 경우 | 우선 검토 대상으로 분류하고 리포트 초안을 확인한다. |

## 검토 상태 정의

| 상태 | 설명 |
|---|---|
| 미확인 | 탐지 결과가 생성되었으나 담당자가 아직 확인하지 않은 상태 |
| 검토중 | 담당자가 거래 상세와 리포트를 확인하고 있는 상태 |
| 정상거래 | 담당자 검토 결과 정상 거래로 판단한 상태 |
| 의심거래 | 담당자 검토 결과 의심거래로 판단한 상태 |

## 샘플 데이터 시나리오

- 정상 소액 이체: 기존 기기와 통상 채널에서 발생한 낮은 금액의 거래
- 고액 이체: 기준 금액 이상으로 발생한 이체
- 신규 기기 거래: 기존 거래 이력에 없는 `device_id`에서 발생한 거래
- 위치 불일치 거래: 짧은 시간 안에 물리적으로 이동하기 어려운 지역에서 발생한 거래
- 반복 거래: 짧은 시간 내 동일 고객 또는 동일 계좌에서 반복 발생한 거래
- 심야 거래: 00:00~05:00 사이에 발생한 거래

## 향후 real 데이터 확장 시 고려사항

- 실제 데이터 연동 시 개인정보와 금융거래정보는 저장, 조회, 로그 출력 범위를 별도로 통제해야 한다.
- `transaction_id`, `customer_id`, `account_id`는 내부 식별 체계와 매핑할 수 있도록 별도 익명화 또는 토큰화가 필요하다.
- 실시간 FDS 운영계와 연동할 경우, 분석 요청과 탐지 결과 생성 시점의 지연시간 기준을 정의해야 한다.
- 탐지 룰은 운영 정책에 따라 버전 관리되어야 하며, 룰 변경 이력과 탐지 결과의 재현 가능성을 보장해야 한다.
- AI 리포트 생성에는 필요한 최소 필드만 전달하고, API Key 또는 민감정보가 프론트엔드나 문서에 노출되지 않도록 관리해야 한다.
