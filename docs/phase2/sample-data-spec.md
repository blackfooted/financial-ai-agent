# Phase 2 샘플 데이터 설계서

## 문서 목적

본 문서는 Phase 2 샘플 거래 데이터의 규모, 케이스 구성, 기대 탐지 결과를 정의한다. Step 3 백엔드 mock API 구현 시 정상 거래와 의심 후보 거래가 균형 있게 포함되도록 기준을 제공한다.

## 샘플 데이터 설계 원칙

- 최소 20건의 거래 데이터를 포함한다.
- 정상 거래와 의심 후보 거래를 함께 포함한다.
- `high(상)`, `medium(중)`, `low(하)` 케이스가 모두 포함되도록 한다.
- 미탐지 정상 거래도 포함한다.
- 실제 고객정보, 계좌번호, IP, 기기정보는 사용하지 않고 가상의 값만 사용한다.
- `review_status` 초기값은 기본적으로 `미확인`으로 둔다.

## 샘플 데이터 규모

| 항목 | 기준 |
|---|---|
| 전체 거래 수 | 최소 20건 |
| 정상 거래 | 미탐지 정상 거래 포함 |
| 의심 후보 거래 | 단일 룰 및 복합 룰 케이스 포함 |
| 위험도 분포 | low(하), medium(중), high(상) 모두 포함 |
| 검토 상태 초기값 | 미확인 |

## 샘플 거래 케이스 목록

| 케이스 ID | 거래 시나리오 | 설명 |
|---|---|---|
| CASE-001 | 정상 소액 이체 | 기존 기기와 일반 시간대에 발생한 소액 이체 |
| CASE-002 | 고액 이체 | 3,000,000원 이상 이체 |
| CASE-003 | 신규 기기 거래 | 기존 이력에 없는 기기에서 발생한 거래 |
| CASE-004 | 반복 거래 | 10분 이내 3회 이상 발생한 거래 |
| CASE-005 | 심야 거래 | 00:00~05:00 사이 발생한 거래 |
| CASE-006 | 위치 불일치 거래 | 1시간 이내 이동 불가능한 지역에서 발생한 거래 |
| CASE-007 | 고액 + 신규 기기 복합 거래 | 고액 이체와 신규 기기 사용이 함께 발생한 거래 |
| CASE-008 | 반복 + 심야 복합 거래 | 반복 거래와 심야 거래가 함께 발생한 거래 |
| CASE-009 | 고액 + 신규 기기 + 위치 불일치 | 주요 위험 신호가 복합적으로 발생한 거래 |
| CASE-010 | 정상 카드 결제 | 일반 채널과 기존 기기에서 발생한 낮은 금액의 결제 |

## 20건 구성 예시

| 구분 | 권장 건수 | 목적 |
|---|---|---|
| 미탐지 정상 거래 | 6건 | 기본 목록 조회와 정상 케이스 검증 |
| 단일 룰 탐지 거래 | 7건 | 룰별 탐지 여부 검증 |
| 복합 룰 medium 거래 | 5건 | risk_score와 medium(중) 매핑 검증 |
| 복합 룰 high 거래 | 2건 | 우선 검토 대상 표시 검증 |

## 화면/API 검증 관점

- 거래 목록에서 `risk_level` 코드값과 화면 표시명 `하`, `중`, `상`이 일관되게 표시되는지 확인한다.
- 상세 조회에서 `detected_rules`, `risk_score`, `reason_summary`, `review_status`가 함께 확인되는지 검증한다.
- PATCH 상태 변경 API에서 `미확인`, `검토중`, `정상거래`, `의심거래` 상태가 모두 처리 가능한지 검증한다.
- 리포트 조회 API에서 룰 기반 탐지 결과가 담당자 검토용 문장으로 설명되는지 확인한다.
- mock 모드에서 외부 API 호출 없이 동일한 기대 결과가 재현되는지 확인한다.

## 향후 실제 데이터 전환 시 고려사항

- 실제 거래 데이터는 개인정보와 금융거래정보 보호 기준에 따라 익명화 또는 마스킹해야 한다.
- 샘플 기준값은 운영 정책과 고객별 거래 패턴에 맞게 재조정해야 한다.
- 실제 운영계에서는 로그인, 권한, 감사 로그, 상태 변경 이력 보존이 필요하다.
- 룰 기반 탐지 결과와 AI 리포트는 최종 판단이 아니며 담당자 검토 근거로만 사용해야 한다.
