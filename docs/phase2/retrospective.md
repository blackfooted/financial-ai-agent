# Phase 2 회고록

## 1. 잘된 점

- Phase 1 회고를 반영해 문서 작성, 탐지 룰 정의, 백엔드 mock API, 프론트 화면 연결, 비용 방어 기준 순서로 진행했다.
- Phase 1 보호 기준을 명확히 두고 Phase 2를 분리 구현해 기존 추천 화면과 API에 영향을 주지 않았다.
- mock 기반으로 외부 API 비용 없이 거래 분석, 목록 조회, 상세 조회, 리포트 조회, 검토 상태 변경까지 E2E 흐름을 검증했다.
- FDS 업무 구조를 룰 기반 탐지와 담당자 검토 흐름으로 표현했다.
- AI가 최종 판단하지 않는 Human-in-the-loop 원칙을 문서와 화면 흐름에 반영했다.
- 포트폴리오 시연 가능한 `/phase2` 화면과 Phase 2 API 흐름을 확보했다.

## 2. 아쉬운 점

- 실제 금융 데이터가 아니라 샘플 거래 데이터를 기반으로 하므로 운영 환경의 데이터 다양성과 복잡도를 충분히 반영하지 못한다.
- 탐지 룰 기준이 운영 FDS 수준보다 단순화되어 있으며, 정교한 이상 탐지 모델이나 통계 기반 기준은 포함하지 않았다.
- 로그인, 권한, 감사 로그, 상태 변경 이력 관리는 제외되어 실제 운영계 검토 프로세스와 차이가 있다.
- OpenAI 실제 호출은 아직 연결하지 않았고, mock 템플릿 기반 리포트로 검증했다.
- 메모리 기반 상태 관리이므로 서버 재시작 시 검토 상태가 초기화된다.

## 3. 다음 Phase 반영사항

- OpenAI 실제 호출 전 `PHASE2_DAILY_LIMIT` 기반 비용 제한 로직을 더 정교화한다.
- 리포트 프롬프트 템플릿과 mock 템플릿을 별도 파일 또는 계층으로 분리하는 방안을 검토한다.
- 처리 이력 또는 감사 로그 구조를 보강해 담당자 검토 흐름의 추적성을 높인다.
- Render 배포 후 cold start, API 응답, 화면 동작에 대한 운영 검증 기록을 추가한다.
- Phase 3 또는 후속 단계에서는 민원 분류, 자동 응답, 기업 분석 등 실제 업무 도메인 확장을 고려한다.
- 메모리 기반 상태 관리의 한계를 보완하기 위해 데이터 저장소 도입 여부를 검토한다.
- 포트폴리오용 화면 설명, 데모 시나리오, 수동 검증 스크립트를 보강한다.
