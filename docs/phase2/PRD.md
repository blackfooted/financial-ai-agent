# Phase 2 PRD

## 문서 목적

본 문서는 Phase 2 프로젝트인 "FDS 의심거래 탐지 및 담당자 검토 리포트 자동화 에이전트"의 제품 요구사항을 정의한다. 개발 착수 전 업무 목표, 사용자 흐름, 기능 범위, 제외 범위, 우선순위를 명확히 하여 이후 데이터 설계, API 설계, 구현 작업의 기준 문서로 활용한다.

## 프로젝트 개요

Phase 2는 금융권 FDS(Fraud Detection System) 업무 맥락을 반영한 포트폴리오형 업무 자동화 에이전트이다. 샘플 거래 데이터를 기반으로 의심 패턴을 탐지하고, 탐지된 후보 거래에 대해 AI가 담당자 검토용 리포트 초안을 생성한다.

본 프로젝트는 자동 차단 시스템이 아니라, 담당자가 빠르게 거래 맥락을 이해하고 검토 판단을 내릴 수 있도록 보조하는 Human-in-the-loop 기반 업무 지원 도구를 지향한다.

## Phase 2 목표

- 샘플 거래 데이터에서 이상 거래 후보를 조회하고 분석한다.
- 룰 기반 탐지 기준으로 의심거래 후보를 식별한다.
- 거래별 위험도와 탐지 사유를 담당자가 이해할 수 있는 형태로 제공한다.
- 위험도 값은 API와 데이터에서는 `low`, `medium`, `high`를 사용하고, 화면 표시명은 `low(하)`, `medium(중)`, `high(상)`으로 표기한다.
- AI를 활용해 검토 리포트 초안을 생성하되, 최종 판단은 담당자가 수행한다.
- mock 모드를 기본 동작으로 유지하여 비용과 외부 API 의존성을 통제한다.

## 사용자 시나리오

1. 담당자는 Phase 2 화면 또는 API를 통해 샘플 거래 목록을 조회한다.
2. 담당자는 `requested_by` 값을 포함해 거래 분석을 요청한다.
3. 시스템은 사전 정의된 룰을 기준으로 의심거래 후보를 탐지한다.
4. 시스템은 거래별 위험도, 탐지 규칙, 요약 사유를 생성한다.
5. 담당자는 탐지 결과를 확인하고 검토 상태를 `미확인`, `검토중`, `정상거래`, `의심거래` 중 하나로 변경한다.
6. 상태 변경 시 담당자 식별자는 인증 세션이 아니라 요청 파라미터의 `reviewed_by` 값으로 전달한다.
7. 담당자는 특정 거래의 AI 리포트 초안을 조회한다.
8. 담당자는 리포트를 참고하되, 정상거래 또는 의심거래 여부를 직접 판단한다.

## 핵심 기능 범위

- 샘플 거래 데이터 목록 조회
- 거래 상세 조회
- 거래 데이터 분석 요청
- 룰 기반 의심거래 탐지
- `risk_score` 산정 및 `low(하)`, `medium(중)`, `high(상)` 위험도 분류
- 담당자 검토 상태 관리
- AI 리포트 초안 생성 및 조회
- 처리 이력 관리
- mock 기반 기본 동작
- openai 모드 전환 시 비용 방어 기준 적용

## 제외 범위

- 실제 금융기관 계정계, 채널계, FDS 운영계 연동
- 실시간 거래 차단 또는 지급 정지
- 고객 통지, 상담 시스템, 민원 시스템 연동
- 실제 개인식별정보 또는 민감정보 저장
- 머신러닝 모델 학습 및 배포
- 로그인, 인증, 권한 관리 기능
- 인증 세션 기반 담당자 식별
- 운영자 권한 관리, 감사 로그, 전자결재 등 엔터프라이즈 운영 기능
- 실제 OpenAI 호출의 기본 활성화

## 요구사항 정의서

| 번호 | 요구사항명 | 설명 | 우선순위 |
|---|---|---|---|
| RQ-001 | 샘플 거래 데이터 조회 | 담당자가 분석 대상 샘플 거래 목록과 상세 정보를 조회할 수 있어야 한다. | High |
| RQ-002 | 거래 데이터 분석 요청 | 담당자가 `requested_by` 값을 포함해 샘플 거래 데이터 분석을 명시적으로 요청할 수 있어야 한다. | High |
| RQ-003 | 룰 기반 의심거래 탐지 | 시스템은 고액 거래, 신규 기기, 반복 거래, 심야 거래, 위치 불일치 등 사전 정의된 룰을 기준으로 의심 후보를 탐지해야 한다. | High |
| RQ-004 | 위험도 산정 | 시스템은 탐지된 거래에 대해 `low`, `medium`, `high` 코드값과 화면 표시명 `하`, `중`, `상` 기준의 위험도를 산정해야 한다. | High |
| RQ-005 | 담당자 검토 상태 관리 | 담당자는 거래별 검토 상태를 `미확인`, `검토중`, `정상거래`, `의심거래` 중 하나로 관리할 수 있어야 한다. | High |
| RQ-006 | AI 리포트 초안 생성 | 시스템은 탐지 결과를 바탕으로 담당자 검토용 리포트 초안을 생성해야 한다. | High |
| RQ-007 | 리포트 조회 | 담당자는 특정 거래의 리포트 초안을 조회할 수 있어야 한다. | High |
| RQ-008 | 처리 이력 관리 | 시스템은 분석 요청, 상태 변경, 리포트 생성 등 주요 처리 이력을 추적할 수 있어야 한다. | High |
| RQ-009 | mock 기본 동작 | Phase 2의 기본 AI Provider는 mock으로 동작해야 하며, 외부 API 호출 없이 기능 흐름을 검증할 수 있어야 한다. | High |
| RQ-010 | 비용 방어 기준 | openai 모드 전환 시 일일 호출 제한, 명시적 승인, 입력 데이터 최소화 등 비용 방어 기준을 적용해야 한다. | High |
| RQ-011 | Human-in-the-loop 원칙 | AI는 거래를 직접 차단하거나 최종 판단하지 않으며, 담당자 검토를 보조하는 리포트 초안만 생성해야 한다. | High |
